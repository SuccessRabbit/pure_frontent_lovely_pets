# 萌宠直播公司项目技术文档

## 1. 项目概览

当前项目是一个以画布渲染为核心的前端游戏应用：

- React 负责应用入口和页面分流
- PixiJS 负责主要 2D 游戏界面与交互
- Three.js 负责 3D 等轴宠物和格子底板
- Zustand 保存运行时状态
- `data/*.csv -> config/*` 构成配置编译链路

项目已不是早期文档中的 DOM 卡牌演示页面，技术口径应以现有源码为准。

## 2. 运行入口与主链路

入口调用链：

```text
src/main.tsx
  -> src/App.tsx
     -> /admin 路由：AdminPage
     -> 游戏路由：useConfigHotReload + useCardLoader + GameBoard
        -> src/game/bootstrap.ts
           -> GameEngine
           -> SceneManager
           -> GameScene
```

说明：

- `src/main.tsx` 不启用 `StrictMode`，避免 WebGL 开发期双挂载
- `src/App.tsx` 根据路径切换游戏页或后台页
- `src/pages/GameBoard.tsx` 只挂载一个全屏 `canvas`
- `useCardLoader()` 会初始化牌库并触发 `initGame`
- `useConfigHotReload()` 在本地监听 Admin 变更并刷新页面

## 3. 渲染架构

### 3.1 分层

```text
React
└── canvas host
    └── PixiJS
        ├── 网格
        ├── 手牌
        ├── HUD
        ├── 拖拽交互
        └── 2D 特效

额外透明 canvas
└── Three.js
    ├── 3D 萌宠
    └── 3D 格子底板与高亮
```

### 3.2 关键模块

- `src/game/core/GameEngine.ts`：Pixi `Application` 初始化、resize、ticker、场景驱动
- `src/game/core/SceneManager.ts`：场景注册与切换
- `src/game/scenes/GameScene.ts`：主场景总控
- `src/game/entities/CardSprite.ts`：手牌卡面绘制与交互
- `src/game/entities/GridCell.ts`：2D 网格格子与压力条
- `src/game/renderers/IsometricPetRenderer.ts`：Three 侧 3D 宠物与格子表现
- `src/game/systems/DragSystem.ts`：拖拽、放置、选格判定
- `src/game/systems/VfxQueue.ts`：表现层特效队列

### 3.3 当前特征

- 可见游戏 UI 基本不依赖 DOM 组件树
- `GameScene` 仍然是主逻辑聚合点，承担 UI、同步、回合表现和 3D 桥接
- Pixi 和 Three 双画布并存，换来表现力，也增加同步复杂度

## 4. 状态与规则架构

### 4.1 状态源

`src/store/gameStore.ts` 是运行时状态入口，保存：

- 回合与阶段
- 小罐头、小红心、生命值
- 3x6 网格和工位耐久
- 手牌、牌库、弃牌堆
- 压力、拆家记录
- 收入倍率、待注入卡牌、抽牌事件等 UI/规则辅助状态

### 4.2 规则执行

当前规则并不只写在 store 中，主执行路径已经迁移到：

- `src/game/rules/ResolutionEngine.ts`
- `src/game/rules/commands.ts`
- `src/game/rules/events.ts`
- `src/game/rules/presentation.ts`

`gameStore` 主要通过 `runGameCommand()` 驱动状态推进。

这意味着当前项目已经具备“命令 -> 规则解析 -> 新状态/事件”的雏形，而不是单纯在组件里直接改 Zustand。

### 4.3 已落地的主要规则

- 开局抽牌、每日抽牌、弃牌堆洗回牌库
- 实体卡放置、行动牌打出、费用扣除与失败回退
- 收入阶段结算
- 手上限整理
- 工位废墟与修复
- 回合末压力增加
- 到达压力上限后的黑红暴走/彻底拆家
- 小红心胜利判定、生命值和全废墟失败判定

### 4.4 未完全数据驱动的部分

虽然配置中已有 `skillTemplates`、`cardSkills` 与运行时 `skills` 字段，但当前仍存在这些现实限制：

- 行动牌目标模式仍有兜底硬编码
- 实体被动尚未全面接入通用技能执行管线
- 羁绊配置已存在，但运行时接入仍不完整
- 早期文档中的部分经济规则与现实现状不一致，应以代码和 `global_config.csv` 为准

## 5. 数据与配置链路

### 5.1 编译流程

```text
data/*.csv
  -> scripts/compile-data.js
     -> config/*.json
     -> config/gameRules.ts
     -> src/types/cards.generated.ts
```

### 5.2 配置分层

- `data/*.csv`：人工维护源数据
- `config/cards.json` 等：按卡牌类型拆分后的运行时配置
- `config/runtimeConfig.json`：完整运行时快照
- `config/gameRules.ts`：从全局配置生成的常量
- `src/utils/runtimeConfig.ts`：运行时读取封装

### 5.3 Admin 设计

本地 Admin 由两部分组成：

- `src/admin/AdminPage.tsx`：React 后台编辑界面
- `scripts/admin-api.js`：本地 HTTP + SSE 服务

后台支持：

- 读取全部原始数据集
- 保存 CSV 并重新编译
- 通过 SSE 广播配置更新
- 非本地环境下回退到只读 runtime 数据

## 6. 资源系统

### 6.1 目录职责

- `public/assets/cards/`：配置层默认卡图来源
- `public/assets/illustrations/`：插画兼容目录
- `public/assets/ui/icons/`：UI 图标

### 6.2 加载策略

`src/game/utils/illustrationTextures.ts` 会按候选路径顺序尝试贴图：

1. 卡牌配置里的 `illustrationPath`
2. 卡牌配置里的 `image`
3. `public/assets/cards/...`
4. `public/assets/illustrations/...`

同时兼容 PNG 与 SVG，并做去重回退。

### 6.3 技术影响

- 历史资源兼容性较好
- 配置、资源和渲染层耦合较深
- 新资源若命名不一致，问题可能在运行时才暴露

## 7. 目录职责

核心目录：

- `src/game/`：Pixi/Three 游戏运行时
- `src/store/`：状态入口与动作封装
- `src/game/rules/`：规则命令、事件和解析引擎
- `src/admin/`：本地配置后台
- `src/hooks/`：初始化与热更新钩子
- `src/utils/`：资源、牌库、配置读取等工具
- `config/`：编译产物与运行时配置
- `data/`：CSV 源数据
- `public/assets/`：静态素材
- `scripts/`：数据编译、CSV 工具、Admin API 与开发脚本

## 8. 当前维护建议

如果继续迭代，优先级建议如下：

1. 继续拆分 `GameScene`，降低单文件职责密度
2. 把更多行动牌和实体被动接入统一技能执行管线
3. 明确 `cards/` 与 `illustrations/` 的长期职责，减少资源兼容成本
4. 为规则层补充自动化测试，优先覆盖回合推进、抽牌、压力和拆家

## 9. 当前限制

- 没有正式测试套件
- 本地编辑依赖 `scripts/admin-api.js`
- 双画布渲染存在同步复杂度
- 部分旧文档提到的玩法和当前代码并不完全一致

因此，后续开发与沟通应优先参考以下对象：

- `data/*.csv`
- `config/gameRules.ts`
- `config/runtimeConfig.json`
- `src/store/gameStore.ts`
- `src/game/rules/ResolutionEngine.ts`
- `src/game/scenes/GameScene.ts`
