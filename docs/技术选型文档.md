# 萌宠直播公司 - 技术选型文档

## 一、当前实现摘要

本文档描述的是当前仓库已经落地的技术方案，而不是最初的设计设想。

当前版本的核心特点：

- React 仅负责应用挂载和启动流程，不再承担主要游戏 UI 渲染。
- 2D 游戏界面、手牌、HUD、拖拽和大部分交互由 PixiJS 负责。
- 3D 等轴萌宠和格子底板由 Three.js 负责，并通过透明 canvas 叠在 Pixi 下层。
- 运行时状态统一由 Zustand `gameStore` 管理。
- 卡牌、规则和羁绊等数据仍然使用 `config/*.json` 配置驱动。

需要特别说明的实现现状：

- `config/synergies.json` 已存在，但当前运行时尚未接入羁绊结算。
- 多数宠物/牛马被动效果仍停留在配置文本层，尚未形成通用效果系统。
- 卡牌配置中的 `stress` 当前主要用于卡面显示，尚未作为每张卡自己的自然压力增长值接入。

## 二、技术栈

### 2.1 前端框架

**选择：React 18 + TypeScript**

**当前职责：**

- 启动应用入口
- 初始化牌库加载
- 挂载 `GameBoard` 画布节点
- 处理引擎生命周期

**当前实现特征：**

- `src/main.tsx` 未启用 `StrictMode`，避免 WebGL 上下文在开发模式下双挂载。
- `src/App.tsx` 只负责调用 `useCardLoader()` 并挂载 `GameBoard`。
- `src/pages/GameBoard.tsx` 当前只渲染一个 Pixi 用的 `<canvas>` 根节点。

### 2.2 游戏渲染架构

**选择：PixiJS 8 + Three.js 混合渲染**

这与项目早期“React + CSS + Canvas 特效层”的方案不同。当前仓库已经切换到以画布为中心的实现。

#### 当前分层

```text
React
└── GameBoard（挂载 Pixi canvas）
    └── PixiJS
        ├── 手牌 CardSprite
        ├── 网格 GridCell
        ├── HUD / 阶段条 / 飘字 / 拖拽反馈
        └── 输入与 2D 特效

额外透明 canvas
└── Three.js
    ├── 等轴低多边形萌宠
    └── 3D 格子底板 / 高亮 / 压力锚点
```

#### 选择原因

- PixiJS 适合卡牌、HUD、拖拽、高亮、文字和 2D 特效。
- Three.js 适合当前已经落地的低多边形萌宠和等轴视角空间表达。
- 双画布方案让 2D 交互和 3D 角色表现可以分层推进，而不必把所有内容都塞进单一渲染器。

#### 当前代价

- 需要同步两层 canvas 的尺寸、缩放和偏移。
- Grid 状态变化要同时驱动 Pixi 和 Three 两套显示对象。
- 输入命中、压力锚点、格子高亮存在跨渲染层协同复杂度。

### 2.3 状态管理

**选择：Zustand**

当前 `src/store/gameStore.ts` 作为唯一运行时状态源，负责：

- 回合阶段流转
- 小罐头、小红心、生命值、工位耐久
- 牌库、手牌、弃牌堆
- 场上实体数据
- 行动牌打出与结算
- 压力累积与拆家逻辑

#### 当前状态结构摘要

```typescript
interface GameState {
  turn: number;
  phase: 'preparation' | 'action' | 'income' | 'end';

  cans: number;
  interest: number;
  winStreak: number;
  loseStreak: number;

  hearts: number;
  playerHp: number;
  maxPlayerHp: number;

  cellDurability: number[][];
  grid: (GridEntity | null)[][];

  hand: Card[];
  deck: Card[];
  discardPile: Card[];

  globalStress: number;
  meltdownHistory: Array<{
    turn: number;
    entityId: string;
    success: boolean;
  }>;

  petIncomeMultiplierThisTurn: number;
  workerIncomeMultiplierThisTurn: number;
  pendingCardsNextTurnDiscard: Card[];
  awaitingHandTrim: boolean;
}
```

### 2.4 构建工具

**选择：Vite**

当前用途：

- TypeScript + React 开发构建
- JSON 配置直接导入
- 快速本地热更新
- 生产构建输出 `dist/`

### 2.5 样式方案

**当前组合：Tailwind CSS + 全局 CSS + Canvas 内绘制**

需要澄清的是：

- Tailwind 已配置并在 `src/index.css` 中注入基础层。
- 当前主要游戏 UI 并不是用 Tailwind 组件搭出来的，而是由 Pixi 直接绘制。
- 当前项目没有成体系使用 CSS Modules；视觉主体已经迁移到画布层。

### 2.6 3D 技术栈

**选择：Three.js**

当前职责：

- 渲染等轴低多边形萌宠
- 渲染 3D 格子底板和高亮状态
- 根据运行时压力值切换宠物动画状态
- 向 Pixi 提供压力条锚点和网格命中辅助

### 3.3 当前职责边界

- `GameEngine`：管理 Pixi `Application`、画布缩放、输入管理、场景切换。
- `GameScene`：当前的主场景，负责 HUD、网格、手牌、拖拽、自动回合推进、Three.js 画布桥接。
- `CardSprite`：手牌卡牌的视觉与交互实体。
- `GridCell`：2D 网格格子和压力条显示。
- `IsometricPetRenderer`：Three.js 侧的 3D 宠物和格子渲染。
- `DragSystem`：拖拽、放置、高亮、行动牌选格等交互协调。
- `gameStore`：规则与运行时状态源。
- `actionEffects.ts`：行动牌按 `card.id` 的具体结算实现。

## 五、资源加载策略

### 5.1 当前资源组织

```text
public/assets/
├── cards/
│   ├── pets/
│   ├── workers/
│   └── actions/
├── illustrations/
│   ├── pets/
│   ├── workers/
│   └── actions/
└── ui/
    └── icons/
```

### 5.2 当前贴图加载顺序

手牌卡牌的插画加载逻辑位于 `src/game/utils/illustrationTextures.ts`，会按顺序尝试：

1. 配置里的 `card.image`
2. `card.image` 对应的同路径 `.svg`
3. `/assets/cards/<folder>/<id>.png`
4. `/assets/cards/<folder>/<id>.svg`
5. `/assets/illustrations/<folder>/<id>.png`
6. `/assets/illustrations/<folder>/<id>.svg`

实体上场后，网格中的贴图会尝试：

1. `/assets/cards/<folder>/<id>.png`
2. `/assets/cards/<folder>/<id>.svg`
3. `/assets/illustrations/<folder>/<id>.png`
4. `/assets/illustrations/<folder>/<id>.svg`

### 5.3 当前资源策略判断

这意味着当前项目处于“兼容两套资源来源”的阶段：

- `cards/` 更像可直接显示的卡面资源来源
- `illustrations/` 更像独立插画资源来源
- 运行时优先选择已有资源，不强依赖单一路径

优点：

- 兼容历史资源
- PNG 可逐步替换 SVG
- 不会因为单个目录缺图导致整体无法显示

代价：

- 资源结构存在冗余
- 文档和脚本如果不同步，容易造成认知偏差

## 六、运行时规则与技术边界的对应关系

虽然本文档重点是技术选型，但以下几点直接影响代码架构，需要明确记录：

- 收益、压力、拆家逻辑当前集中在 `gameStore.ts`，并未下沉为独立 rules engine。
- 行动牌效果当前是“按卡牌 ID 硬编码分发”，位于 `actionEffects.ts`。
- 配置中的 `effects[]` 还不是完整的可解释 DSL。
- 羁绊配置目前只在数据层存在，没有形成运行时应用管线。
- 多数实体被动效果目前仍是配置文本，不是可执行规则。

这意味着当前技术架构更接近：

```text
配置驱动数据
+ 少量规则常量
+ 手写规则分支
+ 场景侧驱动表现
```

而不是完全的数据驱动规则系统。

## 八、当前风险与技术债

| 项目 | 当前情况 | 风险 |
|------|----------|------|
| `GameScene` 体量过大 | 集中了 UI、回合推进、拖拽桥接、Three 同步 | 后续维护成本高 |
| 规则实现分散 | `gameStore.ts` + `actionEffects.ts` + 场景联动 | 容易出现文档与代码脱节 |
| 羁绊未接入运行时 | `synergies.json` 仅作为配置存在 | 策划数据与实际表现不一致 |
| 被动效果未通用化 | 多数只停留在 `effects` 描述文本 | 新卡扩展成本高 |
| 双 canvas 架构 | Pixi 与 Three 需同步缩放和层级 | 输入与显示联动复杂 |
| 资源路径双轨 | `cards/` 与 `illustrations/` 并存 | 新资源接入容易混乱 |

## 九、结论

当前项目的技术选型已经从“React 页面式卡牌展示”演进为“React 挂载 + Pixi 2D 主界面 + Three 3D 宠物层 + Zustand 状态源”的混合画布架构。

这套方案的优点是：

- 已经具备较强的表现力和可扩展空间
- 2D UI 与 3D 角色表现可以分别演进
- 规则状态集中，便于后续补完系统

当前最优先的技术方向不是继续更换引擎，而是：

1. 把现有规则实现补齐到文档与配置声明一致。
2. 抽离 `GameScene` 和规则模块，降低耦合。
3. 收敛资源目录和加载策略，减少双轨历史包袱。
