# 策划与系统设计指南

## 核心概念

后台的技能系统分为两层：

- 技能模板 `skillTemplates`：定义一种可复用的技能结构，包括触发时机、目标模式、参数 schema、operations 和文案模板。
- 卡牌技能绑定 `cardSkills`：把某个模板绑定到某张卡，并填写这张卡专属的参数。

模板负责“能力类型”，绑定负责“这张卡的具体数值”。

## 技能模板流程

1. 进入 `模板工作台`。
2. 点击 `新建模板`，或选择已有模板修改。
3. 在 `1. 模板类型` 中选择业务意图，也就是 effectKind 对应的 recipe。
4. 填写模板 ID、模板名称和内部说明。
5. 在 `2. 触发与目标` 中确认触发时机、目标模式和可绑定卡牌类型。
6. 在 `3. 参数配置` 中确认参数显示名和默认值。
7. 在 `4. 预览与检查` 中查看文案、引用影响面和校验结果。
8. 保存前处理阻断问题。

## 触发时机

触发时机决定技能何时进入结算。

- `on_play`：卡牌打出时触发，适合大多数行动牌。
- `turn_start`：回合开始触发，适合回合初效果。
- `turn_end`：回合结束触发，适合持续压力或收尾结算。
- `income_calc`：收益计算时触发，适合收益倍率和被动收益修正。
- `before_stress_apply`：压力施加前触发，适合免疫或改写压力。
- `before_meltdown`：拆家前触发，适合阻止或改变拆家。
- `after_meltdown`：拆家后触发，适合补偿或后续惩罚。
- `passive`：纯被动说明，不主动驱动运行时事件。

## 目标模式

目标模式决定玩家或系统如何选择作用对象。

- `none`：无需显式目标，打出后直接生效。
- `self`：作用于自身。
- `pet`：玩家选择单个萌宠。
- `worker`：玩家选择单个员工。
- `adjacent_pets`：自动作用于相邻萌宠。
- `adjacent_workers`：自动作用于相邻员工。
- `swap`：需要两个目标，用于位置交换类效果。

如果目标模式需要第二目标，应确认 `supportsSecondTarget` 与目标模式一致。

## 相邻范围判定

当前运行时的“相邻”是正交四方向，不是周围 8 格。

- 上：`row - 1, col`
- 下：`row + 1, col`
- 左：`row, col - 1`
- 右：`row, col + 1`

例如 `adjacent_pets`、`adjacent_workers`、`adjust_stress_adjacent` 和 `adjacent_orthogonal` 都按上下左右四格判断。斜向格子不会被视为相邻目标。

如果设计需要影响周围 8 格，需要新增明确的目标模式或 operation pattern，不能只沿用当前“相邻”文案。

## 参数 schema

参数 schema 定义运营在卡牌绑定里能填写哪些字段。字段名是模板内部变量，字段显示名是后台表单给运营看的名称。

设计参数时遵循这些规则：

- 参数名要稳定，避免上线后频繁改名。
- 数值参数设置合理默认值、最小值和最大值。
- 下拉参数优先使用固定选项，减少运营输入错误。
- 文案模板中使用的 `{变量名}` 必须存在于参数 schema 中。

例如模板文案 `本回合 {entityType} 收益 x{multiplier}` 要求 schema 中存在 `entityType` 和 `multiplier`。

## Operations

operations 是运行时实际执行链。普通情况下优先使用 recipe 自动生成的默认链，不建议手工改动。

高级模式适合以下情况：

- 需要排查某个模板的运行链是否符合预期。
- 已有 recipe 无法覆盖某个少量特殊效果。
- 研发已经确认某个 operation kind 可用，并要求策划配置。

修改 operations 后必须检查 JSON 合法性、selector、filters 和 params。保存前校验会阻断非法 JSON 或缺少 kind 的 operation。

## Scope 与卡牌绑定

`scope` 决定模板能绑定到哪些卡牌类型。卡牌页的模板下拉只显示支持当前卡牌类型的模板。

常见卡牌类型：

- `entity_pet`：萌宠实体。
- `entity_worker`：员工实体。
- `action_buff`：增益行动牌。
- `action_debuff`：减益行动牌。
- `action_utility`：功能行动牌。

如果运营反馈某张卡选不到模板，优先检查模板 `scope` 是否包含该卡类型。

## 校验规则

保存前会进行前端结构校验。阻断问题必须修复后才能保存，警告可以保存但建议处理。

常见阻断问题：

- 模板 ID 为空或重复。
- 模板名称为空。
- 触发时机或目标模式为空。
- 参数字段缺少 name 或 label。
- select 参数没有选项。
- operation 缺少 kind。
- operation 的 filtersJson 或 paramsJson 不是合法 JSON。
- 文案模板引用了未定义变量。
- 技能绑定引用了不存在的模板或卡牌。
- 技能绑定缺少必填参数。
- 模板 scope 不支持已绑定卡牌类型。

常见警告：

- effectKind 未注册 recipe。
- 触发时机或目标模式不在当前已知范围。
- recipe 推荐 scope 与当前 scope 不一致。

## 设计建议

- 先抽象模板，再绑定卡牌；不要为了每张卡复制一套几乎相同的模板。
- 同一类技能只改数值时，应复用模板并通过绑定参数区分。
- 影响目标选择、触发时机或执行链时，才考虑新增模板。
- 新模板上线前至少绑定一张测试卡，在右侧 `Skill Flow` 和游戏内各验证一次。
