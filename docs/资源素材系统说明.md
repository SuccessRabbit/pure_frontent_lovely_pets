# 资源素材系统说明

## 一、当前实现摘要

当前项目的资源系统已经不是“单一路径 + 单一格式”的简单方案，而是一个兼容历史素材与新素材的多候选加载体系。

需要先明确三点：

- 当前运行时主要由 PixiJS 渲染卡牌、HUD、拖拽和 2D 特效。
- 卡牌视觉是代码绘制的，不是直接把整张卡图原样贴上去。
- 仓库中同时保留了 `public/assets/cards/` 和 `public/assets/illustrations/` 两套资源目录，运行时会按顺序尝试加载。

---

## 二、当前资源目录结构

```text
public/assets/
├── cards/
│   ├── pets/                  # 可直接用于卡牌/实体显示的卡面资源
│   ├── workers/
│   ├── actions/
│   └── README.md
├── illustrations/
│   ├── pets/                  # 独立插画资源
│   ├── workers/
│   └── actions/
├── ui/
│   ├── icons/
│   └── README.md
└── effects/
```

### 2.1 两套卡图目录的现状

当前仓库中：

- `cards/` 目录已包含宠物、牛马、行动牌的 SVG 资源。
- `illustrations/` 目录也包含对应分类的 SVG 资源。
- 配置文件中的 `image` 字段默认指向 `cards/` 目录。
- 运行时仍然支持继续回退到 `illustrations/`。

这说明项目当前仍处于历史资源兼容期，而不是只依赖一套最终目录。

## 三、当前卡牌视觉的真实组成

### 3.1 手牌卡牌当前包含的元素

当前 `src/game/entities/CardSprite.ts` 会动态绘制：

1. 白底圆角卡牌背景
2. 基于稀有度的描边颜色
3. 顶部插画显示区域
4. 右上角费用圆形徽章
5. 卡牌名称
6. 卡牌描述文字
7. 收益文本（若配置了 `income`）
8. 压力文本（若配置了 `stress`）

### 3.2 当前没有绘制的元素

旧文档里提到的一些内容目前并未在 `CardSprite` 中实现：

- 左上角稀有度文字标签
- 标签列表展示
- 独立卡牌框贴图
- 专门的属性图标贴图
- `压力上限` 数值显示

也就是说，当前卡牌更接近“代码绘制底板 + 插画贴图 + 文本信息”，而不是“完整卡牌模板系统”。

### 3.3 网格中的实体显示

场上单位并不复用手牌卡面，而是：

- Pixi `GridCell` 绘制格子、压力条、HUD 文字
- Three.js `IsometricPetRenderer` 绘制 3D 宠物和格子底板

因此，资源素材不只服务于手牌，还服务于：

- 场上实体展示
- HUD 图标
- 飘字和特效
- 未来可能接入的独立插画展示

## 四、当前贴图加载逻辑

### 4.1 手牌卡牌的加载顺序

位于 `src/game/utils/illustrationTextures.ts` 的 `candidateUrlsFromCard()` 当前会依次尝试：

1. 配置中的 `card.image`
2. `card.image` 同路径改后缀为 `.svg`
3. `/assets/cards/<folder>/<id>.png`
4. `/assets/cards/<folder>/<id>.svg`
5. `/assets/illustrations/<folder>/<id>.png`
6. `/assets/illustrations/<folder>/<id>.svg`

### 4.2 网格实体的加载顺序

`candidateUrlsFromEntity()` 当前会依次尝试：

1. `/assets/cards/<folder>/<id>.png`
2. `/assets/cards/<folder>/<id>.svg`
3. `/assets/illustrations/<folder>/<id>.png`
4. `/assets/illustrations/<folder>/<id>.svg`

### 4.3 当前加载策略的意义

当前策略支持：

- PNG 优先，SVG 兜底
- `cards/` 与 `illustrations/` 双目录兼容
- 单个资源缺失时继续回退

当前策略的代价：

- 资源来源不够单一
- 文档、Excel 脚本、运行时加载顺序必须同时维护
- 新增资源时，如果命名或目录不一致，问题不一定立刻暴露

## 六、当前资源命名与路径约定

### 6.1 命名规则

- 文件名必须与卡牌 ID 一致
- 宠物：`pet_001` 这类命名
- 牛马：`worker_001` 这类命名
- 行动牌：`action_001` 这类命名

### 6.2 类型到目录的映射

```text
entity_pet     -> pets
entity_worker  -> workers
action_*       -> actions
```

### 6.3 配置文件中的默认路径

Excel 转 JSON 脚本当前默认将 `image` 写为：

```text
/assets/cards/<folder>/<id>.svg
```

这意味着：

- `cards/` 目录是当前配置层的默认首选资源目录
- `illustrations/` 目录更像运行时的兼容回退来源

## 七、添加新素材时的建议流程

### 7.1 推荐做法

如果要新增或替换卡图，建议按以下顺序处理：

1. 先保证文件名与卡牌 ID 一致。
2. 优先补到 `public/assets/cards/<type>/`。
3. 如果还需要独立插画用途，再同步补到 `public/assets/illustrations/<type>/`。
4. 若提供 PNG，保持与 SVG 同名，运行时会优先加载 PNG。

### 7.2 推荐尺寸

- 插画区域的设计比例仍建议使用 4:3。
- SVG 建议使用 `viewBox="0 0 400 300"`。
- PNG 建议至少 `400x300`，便于后续缩放。

### 7.3 风格建议

- 宠物：暖色、圆润、表情鲜明
- 牛马：可略偏冷色，但整体仍需保持同一世界观
- 行动牌：符号化强、辨识度优先
- UI 图标：形体清晰，便于缩小显示

## 八、当前未完全落地的素材能力

以下能力在代码或文档中被提到过，但当前并未完整接入：

- 稀有度角标贴图
- 独立卡框资源
- 标签视觉系统
- 统一的属性图标渲染体系
- 通过素材驱动的完整卡面模板切换

当前 `src/utils/assetLoader.ts` 中虽然保留了 `getCardFramePath()` 等函数，但仓库内并没有配套使用中的框架资源，也未在 `CardSprite` 中接入。

## 十、结论

当前资源系统的核心特征不是“资源少”，而是“资源路径和展示职责处于兼容过渡态”。

它的优点是：

- 能兼容历史 SVG 卡面和新插画资源
- 可逐步引入 PNG 替换
- 即使部分资源缺失也不容易直接导致黑图

它当前最需要优化的点是：

1. 明确 `cards/` 与 `illustrations/` 的最终职责边界。
2. 让文档、Excel 脚本、运行时加载顺序保持完全一致。
3. 决定是否继续推进“纯代码卡面 + 独立插画”方案，还是收敛到单一资源体系。
